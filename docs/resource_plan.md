# 权限重构方案
- 现在的权限方案太过简单， 只是简单的角色权限， 不能满足复杂的权限需求， 需要重构权限方案。
- 现在的权限实现是这样的，rbac，权限是基于角色的，每个用户有一个角色， 每个角色有一组权限， 但是有些权限是需要细粒度控制的。
- 比如说某个用户只能查看某些资源， 但是不能修改这些资源， 这种情况现在的权限方案是无法实现的。

## 现在的表结构
- sys_user表， 存储用户信息， 包括用户名， 密码， 角色等。
- sys_role表， 存储角色信息， 包括角色名称， 角色描述等
- sys_role_user表， 存储角色和用户的关系， 包括角色id， 用户id等
- sys_menu表， 菜单表，存储菜单，按钮
- sys_role_menu表， 角色和权限的关系， 包括角色id， 权限id等

## 应该需要修改表结构
- sys_menu表，移除菜单类型中的按钮，按钮表允许需要单独一张表，但是也不对！ 按钮应该是跟菜单或者页面强关联的，比如某个菜单下某个目录下有什么按钮？
- 这种关系应该是菜单和按钮的关系， 而不是单独一张表， 因为按钮是跟菜单或者页面强关联的。
- 新增资源 resource表， 存储资源信息， 包括资源名称， 资源类型， 资源路径等,其实就是接口表
- 如果增加了资源表，需要解决的问题之一！ 之前的菜单表方案 是这样的 其实也有接口权限验证，是用的这个has_perms宏，权限方案是使用的jwt，我们使用了双token，但是为了避免token的过期问题，我们使用了refresh token，refresh token的过期时间比较长， access token的过期时间比较短，当access token过期的时候，客户端可以使用refresh token来获取新的access token，这样就避免了token过期的问题。
- 并且为了每次请求不查询redis，将一些数据直接放到了access token中， 但是这样就有一个问题， 权限码太多，导致jwt过于庞大， 影响请求大小，导致请求太大，所以我们后续的方案是使用了位图来存储权限码， 这样就可以大大减少jwt的大小， 但是这样就需要在服务端维护一个权限码和位图的映射关系， 这样就可以通过位图来判断用户是否有某个权限了。
- 如果我们将来增加了一张资源表，难道我们也将资源相关字段进行位图存储，也存储到jwt中？实现起来倒是没有问题！ 不知道是否会很臃肿呢？
