# Admin 能力缺口（待补齐）

本文记录当前 `summerrs-admin` 在“通用后台管理（admin）”场景下，尚缺失或需要加强的一些关键能力点，方便后续规划迭代。

> 说明：这里讨论的是“平台/系统后台”的通用能力，并不等同于业务域（tenant 用户、租户侧 RBAC 等）。若后续要接入“租户用户表/租户权限体系”，这些能力越早补齐越不容易返工。

## 1) 组织架构（部门 / 组织 / 岗位）

当前 `sql/sys` 中未看到典型的 `dept` / `post`（部门 / 岗位）等组织体系相关表与接口，这会影响：

- 用户归属：用户属于哪个部门/组织，是否允许跨部门访问
- 权限分配：角色是否按部门分配，是否支持“部门管理员”
- 数据权限：后续“按部门/组织”过滤的基础

建议补齐的最小模型：

- 部门/组织（树形）：`dept(id, parent_id, name, sort, enabled, ...)`
- 岗位/职级（可选）：`post(id, name, sort, enabled, ...)`
- 关系表：
  - 用户-部门：`user_dept(user_id, dept_id)`（或用户表直接带 `dept_id`）
  - 用户-岗位：`user_post(user_id, post_id)`（可选）

## 2) 数据权限（强烈建议尽早做）

目前更多是“功能权限/RBAC”（用户能不能点某个功能），但很多 admin 项目真正难点在“数据权限”（用户能看到哪些数据）。

常见数据范围模型（同一角色下可配置）：

- 仅本人数据（Self）
- 本部门数据（Dept）
- 本部门及子部门（DeptAndChildren）
- 自定义部门（CustomDepts）
- 全部数据（All）

如果后续要接入“租户用户表/租户业务表”，没有统一的数据权限模型，往往会出现：

- 每个业务接口自己写一套过滤规则（不可维护）
- 需要回头重构查询层/服务层（返工成本高）

建议落地方式（方向性）：

- 在 RBAC 中为角色增加 `data_scope`（枚举）与 `role_dept`（自定义部门列表）
- 在查询层引入统一的“数据范围条件拼接/注入”机制（例如：service 层统一传入 `LoginUser`，或在 repository/query builder 层注入）

## 3) 权限细粒度（按钮 / 接口 / 资源）

当前权限更偏向“菜单维度”（例如菜单上的 `auth_mark` / `auth_list`），但通用 admin 往往还需要更细的统一模型：

- 按钮权限（前端控制可见/可点）
- 接口权限（后端必须校验，避免前端绕过）
- 资源权限（更抽象：例如“文件/对象/项目/订单”的 owner 或 scope）

建议补齐的最小能力：

- “权限点（permission）”作为一等公民：`permission(code, name, type, ...)`
- 角色-权限点绑定：`role_permission(role_id, permission_id)`
- 接口侧统一校验：支持按 `permission:code` 校验，而不是只靠菜单字段

## 4) 账号安全能力（“能用”到“可对抗”）

目前登录/鉴权链路已可用，但距离“抗攻击/可审计”的生产级后台，通常还需要补齐：

- 验证码（登录/敏感操作）
- 登录失败策略：错误次数限制、冷却时间、账号/设备/IP 维度的锁定
- MFA（二次验证）：TOTP/短信/邮箱（至少预留扩展点）
- 密码策略：强度要求、过期、历史密码禁止复用、首次登录强制改密
- 会话安全：踢下线、刷新 token 策略、会话吊销原因记录（用于审计与排障）

建议方向（不强制一次做完）：

- 先落地“失败次数限制 + 冷却”与“验证码”
- 再补“会话吊销 + 踢下线原因”
- 最后再做 “MFA / 密码策略体系”
